# 07 — Snapshots & Migrations

**Status:** Proposed
**Objective:** 2 (Snapshots) and 3 (Migrations) of 3
**Depends on:** Entity Classes (`06-entity-classes.md`)

## Overview

Snapshots capture the full schema state as JSON. Migrations are generated by diffing two snapshots and producing ordered ALTER/CREATE/DROP SQL. A database table tracks which migrations have been applied.

```
DDL files (source of truth)
  → parse → entity classes → snapshot (JSON)

snapshot(N) ←diff→ snapshot(N+1) → migration SQL

migration SQL → apply to database → update _dbd_migrations table
```

## Objective 2: Snapshots

### Command

```bash
dbd snapshot                    # Create next sequential snapshot
dbd snapshot --name "add users" # Create with description
dbd snapshot --list             # List existing snapshots
```

### Snapshot File Structure

```
project/
  snapshots/
    001.json                    # Initial schema
    002.json                    # After adding user tables
    003.json                    # After adding audit triggers
```

### Snapshot Format

```json
{
  "version": 3,
  "description": "Add audit triggers",
  "timestamp": "2026-02-18T10:30:00Z",
  "config": {
    "schemas": ["public", "config", "staging"],
    "extensions": [{ "name": "uuid-ossp", "schema": "public" }]
  },
  "entities": {
    "schemas": [
      { "type": "schema", "name": "config" },
      { "type": "schema", "name": "staging" }
    ],
    "extensions": [{ "type": "extension", "name": "uuid-ossp", "schema": "public" }],
    "roles": [
      { "type": "role", "name": "app_reader", "refers": [] },
      { "type": "role", "name": "app_writer", "refers": ["app_reader"] }
    ],
    "tables": [
      {
        "type": "table",
        "name": "config.profiles",
        "schema": "config",
        "columns": [
          {
            "name": "id",
            "type": "uuid",
            "nullable": false,
            "defaultValue": "uuid_generate_v4()",
            "isPrimaryKey": true
          },
          { "name": "display_name", "type": "varchar(255)", "nullable": true }
        ],
        "constraints": [{ "name": "profiles_pkey", "type": "primary_key", "columns": ["id"] }],
        "indexes": [],
        "refers": []
      }
    ],
    "views": [],
    "functions": [
      {
        "type": "function",
        "name": "config.get_setting",
        "schema": "config",
        "parameters": [{ "name": "key", "type": "text", "mode": "in" }],
        "returnType": "text",
        "language": "plpgsql",
        "body": "BEGIN RETURN ...; END;",
        "refers": ["config.settings"]
      }
    ],
    "procedures": [],
    "triggers": []
  }
}
```

### Snapshot Creation Flow

```
dbd snapshot
  → Design.validate()                  # Normal validation pipeline
  → entities.map(e => e.toJSON())      # Serialize all entity classes
  → determine next version number      # Max existing + 1
  → write snapshots/{version}.json
```

### Version Numbering

- Sequential integers: `001`, `002`, `003`, ...
- Zero-padded to 3 digits (supports up to 999 snapshots; can extend)
- Version is determined by scanning `snapshots/` directory for highest existing number
- No gaps allowed — if 001 and 003 exist but 002 doesn't, that's an error

## Objective 3: Migrations

### Commands

```bash
dbd migrate                     # Generate migration from last snapshot to current DDL
dbd migrate --apply             # Apply pending migrations to database
dbd migrate --status            # Show current DB version and pending migrations
dbd migrate --to 5              # Apply migrations up to version 5
```

### Migration File Structure

```
project/
  snapshots/
    001.json
    002.json
    003.json
  migrations/
    001-to-002.sql              # Generated ALTER statements
    002-to-003.sql              # Generated ALTER statements
```

### Migration Generation Flow

```
dbd migrate
  1. Read latest snapshot (e.g., snapshots/002.json)
  2. Parse current DDL files → build entity classes
  3. Create in-memory snapshot of current state
  4. Diff snapshot 002 vs current state
  5. Generate migration SQL from diff
  6. Write migrations/002-to-003.sql
  7. Write snapshots/003.json (new snapshot)
```

### Schema Diff Algorithm

```javascript
function diffSnapshots(from, to) {
  const changes = []

  // 1. Added entities (in 'to' but not in 'from')
  // 2. Dropped entities (in 'from' but not in 'to')
  // 3. Modified entities (in both, but diff() returns changes)

  for (const entity of to.entities) {
    const prev = from.findByName(entity.name)
    if (!prev) {
      changes.push({ action: 'create', entity })
    } else {
      const diff = entity.diff(prev)
      if (diff) changes.push({ action: 'alter', entity, diff })
    }
  }

  for (const entity of from.entities) {
    if (!to.findByName(entity.name)) {
      changes.push({ action: 'drop', entity })
    }
  }

  return changes
}
```

### Migration SQL Generation

Each entity type generates its own migration SQL based on the diff:

#### Tables

```sql
-- Added column
ALTER TABLE config.profiles ADD COLUMN avatar_url text;

-- Dropped column
ALTER TABLE config.profiles DROP COLUMN legacy_field;

-- Altered column type
ALTER TABLE config.profiles ALTER COLUMN display_name TYPE varchar(500);

-- Altered column nullable
ALTER TABLE config.profiles ALTER COLUMN email SET NOT NULL;

-- Added constraint
ALTER TABLE config.profiles ADD CONSTRAINT profiles_email_unique UNIQUE (email);

-- Dropped constraint
ALTER TABLE config.profiles DROP CONSTRAINT profiles_old_check;

-- Added index
CREATE INDEX idx_profiles_email ON config.profiles (email);

-- Dropped index
DROP INDEX idx_profiles_legacy;
```

#### Views

```sql
-- Changed definition → full replacement
CREATE OR REPLACE VIEW reporting.user_stats AS
  SELECT user_id, count(*) as login_count
  FROM auth.logins
  GROUP BY user_id;
```

#### Functions / Procedures

```sql
-- Body changed → full replacement
CREATE OR REPLACE FUNCTION config.get_setting(key text)
RETURNS text LANGUAGE plpgsql AS $$
BEGIN
  RETURN (SELECT value FROM config.settings WHERE name = key);
END;
$$;

-- Signature changed → DROP + CREATE
DROP FUNCTION IF EXISTS config.old_function(text);
CREATE FUNCTION config.old_function(key text, default_val text)
RETURNS text ...;
```

#### Triggers

```sql
-- Any change → DROP + CREATE (no ALTER TRIGGER for definition changes)
DROP TRIGGER IF EXISTS audit_trigger ON config.profiles;
CREATE TRIGGER audit_trigger
  AFTER INSERT OR UPDATE ON config.profiles
  FOR EACH ROW EXECUTE FUNCTION audit.log_change();
```

#### Schemas / Extensions / Roles

```sql
-- Only additive (CREATE IF NOT EXISTS)
CREATE SCHEMA IF NOT EXISTS reporting;
CREATE EXTENSION IF NOT EXISTS "pgcrypto" WITH SCHEMA public;
```

### Dependency-Ordered Migration

Migration statements are ordered using the same dependency graph:

1. **CREATE** schemas (no dependencies)
2. **CREATE** extensions
3. **CREATE** new tables (dependency order)
4. **ALTER** existing tables (add columns, constraints)
5. **CREATE/REPLACE** functions and procedures
6. **CREATE/REPLACE** views (may depend on new tables/columns)
7. **CREATE** triggers
8. **DROP** triggers (reverse dependency order)
9. **DROP** views
10. **DROP** functions/procedures
11. **ALTER** tables (drop columns, constraints)
12. **DROP** tables (reverse dependency order)

Creates go dependency-first. Drops go reverse-dependency-first.

### Database Version Tracking

A `_dbd_migrations` table tracks applied migrations:

```sql
CREATE TABLE IF NOT EXISTS _dbd_migrations (
  version     integer PRIMARY KEY,
  applied_at  timestamptz NOT NULL DEFAULT now(),
  description text,
  checksum    text NOT NULL    -- SHA-256 of migration file
);
```

#### `dbd migrate --status`

```
Current database version: 2
Pending migrations:
  002-to-003.sql  (Add audit triggers)
```

#### `dbd migrate --apply`

```
Applying 002-to-003.sql...
  ALTER TABLE config.profiles ADD COLUMN avatar_url text;
  CREATE TRIGGER audit_trigger ...
Migration 003 applied successfully.
```

#### Checksum Verification

Before applying a migration, compare the file's SHA-256 against the stored checksum (if re-applying). This catches accidental edits to already-applied migration files.

### Migration Execution Flow

```
dbd migrate --apply
  1. Connect to database via adapter
  2. Read _dbd_migrations → current version
  3. List migrations/ directory → find pending (> current version)
  4. For each pending migration (in order):
     a. Read migration SQL file
     b. Verify checksum (if version exists in table)
     c. Begin transaction
     d. Execute migration SQL
     e. INSERT INTO _dbd_migrations (version, description, checksum)
     f. Commit transaction
  5. Report applied migrations
```

Each migration runs in a transaction. If it fails, the transaction rolls back and the version stays unchanged.

## Edge Cases

### Renamed Entity

A rename shows up as DROP + CREATE. The migration generator doesn't attempt to detect renames — it's safer to be explicit. Users can edit the generated migration SQL to use `ALTER TABLE ... RENAME` if they prefer.

### Data-Dependent Changes

Some ALTER statements may fail at runtime:

- `ALTER COLUMN SET NOT NULL` — fails if NULLs exist
- `ALTER COLUMN TYPE` — may fail if data can't be cast
- `DROP COLUMN` — may fail if views/functions reference it

The migration generator adds comments for potentially dangerous operations:

```sql
-- WARNING: This may fail if NULL values exist in email column
ALTER TABLE config.profiles ALTER COLUMN email SET NOT NULL;
```

### Manual Migration Edits

Generated migrations are starting points. Users can (and should) edit them to:

- Add data migration steps (backfill values before NOT NULL)
- Change DROP + CREATE to RENAME
- Add custom logic not derivable from schema diff

The checksum in `_dbd_migrations` tracks the file as-applied, not as-generated.

### Empty Diff

If no changes are detected between the last snapshot and current DDL, `dbd migrate` reports "No changes detected" and does not create a migration file.

## Package Location

```
packages/cli/src/
  ├── snapshot.js           # Snapshot creation and reading
  ├── migration.js          # Migration generation and application
  └── index.js              # New commands: snapshot, migrate

packages/db/src/
  ├── entities/             # Entity classes with diff() and toJSON()
  ├── schema-diff.js        # diffSnapshots() — pure function
  └── migration-generator.js # diff → SQL generation — pure functions
```

- **Snapshot I/O** lives in CLI (reads/writes files)
- **Diff logic** lives in `packages/db` (pure functions, no I/O)
- **SQL generation** lives in `packages/db` (pure functions)
- **Migration execution** uses the adapter (CLI orchestrates)
