# 07 — Snapshots & Migrations

**Status:** Implemented
**Depends on:** Entity Classes (`06-entity-classes.md`)

## Overview

Snapshots capture the full table structure as JSON. Migrations are per-table ALTER SQL files generated by diffing consecutive snapshots. A `_dbd_migrations` database table tracks which versions have been applied, scoped by project name to support multiple projects in one database.

```
DDL files (source of truth)
  → parse → entities → snapshot JSON (snapshots/NNN.json)

snapshot(N) ←diff→ snapshot(N+1) → per-table ALTER SQL (migrations/NNN/<schema>/<table>.sql)

dbd apply: pending ALTERs interleaved before each table's DDL → record version in _dbd_migrations
dbd migrate --apply: run ALTERs standalone → record version in _dbd_migrations
```

---

## Snapshot Format

File: `snapshots/NNN.json`

```json
{
  "version": 2,
  "description": "add display_order column",
  "timestamp": "2026-03-30T17:25:48.724Z",
  "tables": [
    {
      "name": "config.lookups",
      "columns": [
        {
          "name": "id",
          "type": "uuid",
          "nullable": false,
          "defaultValue": "uuid_generate_v4()",
          "isPrimaryKey": true
        },
        { "name": "display_order", "type": "integer", "nullable": true, "defaultValue": "0" }
      ],
      "indexes": [{ "name": "lookup_ukey", "columns": ["name"], "unique": true }]
    }
  ]
}
```

Only tables are captured in snapshots. Views, functions, procedures, roles, schemas, and extensions are not diffed.

## Migration Folder Structure

```
migrations/
  002/
    graph.json                  # Metadata: fromVersion, toVersion, altered[], dropped[]
    config/
      lookups.sql               # ALTER SQL for config.lookups
      lookup_values.sql         # ALTER SQL for config.lookup_values
    staging/
      lookup_values.sql         # ALTER SQL for staging.lookup_values
  003/
    graph.json
    config/
      lookup_values.sql
```

The folder structure under each version mirrors the DDL folder structure (`ddl/table/<schema>/<name>.ddl` → `migrations/NNN/<schema>/<name>.sql`).

Dropped tables use `<name>.drop.sql` (e.g. `migrations/003/config/old_table.drop.sql`).

### `graph.json`

```json
{
  "fromVersion": 1,
  "toVersion": 2,
  "altered": ["config.lookups", "staging.lookup_values", "config.lookup_values"],
  "dropped": []
}
```

The `altered` and `dropped` arrays list fully-qualified table names (`schema.table`). They drive both the interleaved apply in `dbd apply` and the standalone apply in `dbd migrate --apply`.

## `_dbd_migrations` Table

```sql
CREATE TABLE IF NOT EXISTS _dbd_migrations (
  project     varchar NOT NULL,
  version     integer NOT NULL,
  applied_at  timestamptz NOT NULL DEFAULT now(),
  description text,
  checksum    text,
  PRIMARY KEY (project, version)
);
```

- `project` — from `project.name` in `design.yaml`, defaulting to the directory basename
- `version` — snapshot version number
- `checksum` — SHA-256 of `graph.json` for the migration; empty string for fresh-apply records
- `description` — snapshot description or migration label

## `dbd apply` Migration Logic

`dbd apply` (full apply, no `-n`) uses this detection sequence:

1. **Read `_dbd_migrations`** → `dbVersion` (max version for this project, or 0 if none)
2. **Find latest local snapshot** → `latestVersion` from `snapshots/` directory
3. **Compute pending** = all migration folders with `toVersion > dbVersion`

**Fresh DB (dbVersion === 0 and no tables exist):**

- Apply all DDL entities in dependency order (no ALTERs)
- Insert one record at `latestVersion` with `description = "fresh apply at vN"`, empty checksum
- Result: `_dbd_migrations` has exactly one row

**Pre-snapshot DB or existing DB (tables exist):**

- Build a map: `tableName → [migrations in version order]`
- Walk `validEntities` in dependency order:
  - For each `table` entity: run all pending migrations for that table (in version order) before applying DDL
  - Apply DDL entity
- Handle dropped tables (from migration `dropped[]` arrays) after all entities
- Insert one record per applied migration version

`dbd reset` drops all schemas, then runs `DELETE FROM _dbd_migrations WHERE project = '...'`.

### Interleaving example

DB at v1, snapshots up to v3. Dependency order has `config.lookup_values` after `config.lookups`:

```
Applying table: config.categories
Migrating config.lookups (v1 → v2)
Applying table: config.lookups
Migrating config.lookup_values (v1 → v2)
Migrating config.lookup_values (v2 → v3)   ← both pending versions, applied before DDL
Applying table: config.lookup_values
```

This is safe with FK constraints because referenced tables are always applied before tables that reference them.

## `dbd migrate` Standalone

`dbd migrate --apply` runs pending migrations without re-applying DDL:

1. Read `dbVersion` from `_dbd_migrations`
2. Find pending migrations (version folders with `toVersion > dbVersion`)
3. For each migration in version order:
   - For each table in `graph.json altered[]`: read and execute `<schema>/<table>.sql`
   - For each table in `graph.json dropped[]`: read and execute `<schema>/<table>.drop.sql`
   - Insert record into `_dbd_migrations`

`--dry-run` prints SQL without executing. `--to N` limits to versions ≤ N.

## `dbd snapshot` Creation Flow

```
dbd snapshot [--name "description"]
  1. validate entities
  2. extract table structures from entities → current snapshot object
  3. determine next version (max existing + 1, defaulting to 1)
  4. write snapshots/{version}.json
  5. if version === 1: done (no migration)
  6. diff previous snapshot against current snapshot
  7. if empty diff: done (no migration folder)
  8. write migrations/{version}/graph.json
  9. for each altered table: write migrations/{version}/<schema>/<table>.sql
  10. for each dropped table: write migrations/{version}/<schema>/<table>.drop.sql
```

### Diff scope

Only table-level changes are diffed:

- Added columns → `ADD COLUMN`
- Dropped columns → `DROP COLUMN`
- Changed column type or nullable → `ALTER COLUMN TYPE` / `SET NOT NULL` / `DROP NOT NULL`
- Changed column default → `ALTER COLUMN SET DEFAULT` / `DROP DEFAULT`
- Added/dropped unique indexes → `CREATE UNIQUE INDEX` / `DROP INDEX`

Added or dropped tables generate no migration SQL (new tables are created by their DDL; dropped tables need a `<name>.drop.sql`).
