# Snapshots & Migrations

DBD tracks schema evolution through **snapshots** (point-in-time JSON captures) and **migrations** (per-table ALTER SQL generated from snapshot diffs). The `_dbd_migrations` meta table records which versions have been applied to each database.

---

## Concepts

| Term              | Meaning                                                                          |
| ----------------- | -------------------------------------------------------------------------------- |
| **Snapshot**      | JSON file capturing table structure at a point in time (`snapshots/NNN.json`)    |
| **Migration**     | Folder of ALTER SQL files generated by diffing two snapshots (`migrations/NNN/`) |
| **Version**       | Sequential integer (1, 2, 3…) shared by snapshot and migration folders           |
| `_dbd_migrations` | Meta table in the database recording applied versions, scoped by project name    |

The DDL files in `ddl/` are always the **source of truth**. Snapshots record what the schema looked like at each version. Migrations contain the SQL to move the database from the previous version to the current one.

---

## Folder structure

```
myproject/
  snapshots/
    001.json              # Initial schema capture
    002.json              # After adding display_order column
    003.json              # After adding notes column
  migrations/
    002/
      graph.json          # { fromVersion, toVersion, altered[], dropped[] }
      config/
        lookups.sql       # ALTER statements for config.lookups
        lookup_values.sql # ALTER statements for config.lookup_values
      staging/
        lookup_values.sql # ALTER statements for staging.lookup_values
    003/
      graph.json
      config/
        lookup_values.sql
```

`migrations/001/` is never created — version 1 is the initial state, nothing to migrate from.

### `graph.json` format

```json
{
  "fromVersion": 2,
  "toVersion": 3,
  "altered": ["config.lookup_values"],
  "dropped": []
}
```

- `altered` — tables with ADD/ALTER/DROP column or index changes
- `dropped` — tables removed in this version (SQL in `<name>.drop.sql`)

### Per-table SQL files

Mirror the DDL folder structure under the migration version folder. A table at `ddl/table/config/lookups.ddl` has its migration SQL at `migrations/002/config/lookups.sql`.

```sql
-- Migration: version 1 → 2
-- Generated: 2026-03-30T17:25:48.724Z

ALTER TABLE "config"."lookups" ADD COLUMN "display_order" int DEFAULT 0;
```

Dropped tables use `<name>.drop.sql`:

```
migrations/003/config/old_table.drop.sql
```

## Commands

### `dbd snapshot`

Capture the current DDL state as a new versioned snapshot and generate a migration folder.

```sh
dbd snapshot                              # Create next snapshot
dbd snapshot --name "add email column"   # Attach a description
dbd snapshot --list                       # List existing snapshots
```

**Output:**

```
Snapshot 002 created (5 tables).
Migration folder: migrations/002
```

If there are no schema changes since the last snapshot:

```
Snapshot 003 created (5 tables).
No schema changes detected — no migration generated.
```

For the first snapshot (version 1):

```
Snapshot 001 created (5 tables).
First snapshot — no migration generated.
```

### `dbd migrate`

Manage and apply pending migrations independently of `dbd apply`.

```sh
dbd migrate                         # Show status (same as --status)
dbd migrate --status                # Show local version vs DB version
dbd migrate --apply                 # Apply all pending migrations
dbd migrate --apply --dry-run       # Print SQL without executing
dbd migrate --apply --to 3          # Apply up to version 3 only
```

**`--status` output:**

```
Local version:    3
Database version: 2

Pending migrations (1):
  002 → 003  alter: config.lookup_values
```

**`--apply` output:**

```
Applying migration v2 → v3
  Migrating config.lookup_values
  Version 3 recorded.
```

**`--dry-run` output:**

```
-- Migration v2 → v3 (migrations/003)

-- Table: config.lookup_values
ALTER TABLE "config"."lookup_values" ADD COLUMN "notes" text;
```

## How `dbd apply` integrates migrations

`dbd apply` (without `--dry-run` or `-n`) automatically applies pending migrations interleaved with DDL application.

**Detection logic:**

1. Read `_dbd_migrations` to get current DB version.
2. If `dbVersion === 0` and no tables exist → **fresh DB**: apply all DDL, record latest snapshot version, skip all ALTER scripts.
3. If `dbVersion === 0` and tables exist → **pre-snapshot DB**: run pending migrations.
4. If `dbVersion > 0` → run pending migrations.

**Interleaving:**

For each entity in dependency order, if the entity is a table and has pending migrations, the ALTER SQL runs immediately before the `CREATE OR REPLACE` DDL:

```
Applying schema: config
Applying table: config.categories
Migrating config.lookups (v1 → v2)      ← ALTER runs before DDL
Applying table: config.lookups
Migrating config.lookup_values (v1 → v2)
Migrating config.lookup_values (v2 → v3) ← Multiple versions catch up
Applying table: config.lookup_values
```

This ordering is safe with FK constraints: tables referenced by FKs are always created before the tables that reference them.

After all entities: dropped tables are handled, then all applied migration versions are recorded in `_dbd_migrations`.

## `_dbd_migrations` table

Created automatically by `dbd apply` or `dbd migrate --apply` on first use.

```sql
CREATE TABLE IF NOT EXISTS _dbd_migrations (
  project     varchar NOT NULL,
  version     integer NOT NULL,
  applied_at  timestamptz NOT NULL DEFAULT now(),
  description text,
  checksum    text,
  PRIMARY KEY (project, version)
);
```

- `project` — from `project.name` in `design.yaml` (allows multiple projects in one database)
- `version` — snapshot version number
- `checksum` — SHA-256 of `graph.json` for the migration (empty for fresh-apply records)

**After fresh apply (reset + apply):** exactly one row at the latest snapshot version.

**After incremental apply:** one row per applied migration version, plus the initial row.

---

## Multi-project databases

If multiple DBD projects share one database (with distinct table names), each project tracks its own migration history via the `project` column. The project name comes from `design.yaml`:

```yaml
project:
  name: MyProject
```

`dbd reset` clears only the rows for the current project. Other projects' migration history is unaffected.

## Typical schema evolution workflow

```sh
# 1. Edit DDL files (add column, create table, etc.)
dbd inspect                         # Validate

# 2. Capture the change as a new snapshot + migration
dbd snapshot --name "add notes column"

# 3. Review the generated migration SQL
cat migrations/003/config/lookup_values.sql

# 4. Apply to database (migrations interleaved automatically)
dbd apply

# 5. Or apply migrations independently without re-running DDL
dbd migrate --apply
```

---

## Reset and rebuild

```sh
dbd reset                 # Drop all schemas, clear _dbd_migrations for this project
dbd apply                 # Rebuild from DDL; records exactly one row at latest version
```

After reset + apply the database is at the current DDL state. No ALTER scripts are run — the DDL files already reflect the final schema. The single `_dbd_migrations` entry marks the baseline for future migrations.

---

## Edge cases

**Catch-up (multiple pending versions):** `dbd apply` and `dbd migrate --apply` both handle multiple pending versions in order. A DB at v1 with snapshots up to v3 will apply v2 then v3 migrations.

**No migrations folder / pre-snapshot DB:** If the database has tables but no `_dbd_migrations` history, `dbd apply` treats it as a pre-snapshot DB and runs pending migrations. If no migration folders exist, it applies DDL normally (idempotent `CREATE OR REPLACE`).

**Empty diff:** If no table structure changed since the last snapshot, `dbd snapshot` creates a snapshot JSON but no migration folder.

**Manual migration edits:** The per-table SQL files are generated as a starting point. Edit them before running `dbd migrate --apply` to add data migrations, rename operations, or other custom SQL. The checksum in `_dbd_migrations` reflects the SQL as applied.
