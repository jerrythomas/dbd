# Convex Support Design

**Date:** 2026-03-28
**Status:** Approved

## Overview

Add support for targeting Convex deployments from dbd. Given an existing PostgreSQL DDL folder, users can generate a `convex/schema.ts` and seed data into a Convex deployment — either via `--target=convex` flags on existing commands or via new `dbd convex` subcommands.

DDL parsing remains unchanged (postgres adapter). Convex is an output target, not a database source.

---

## Architecture

### New package: `packages/convex` (`@jerrythomas/dbd-convex`)

```
packages/convex/
  src/
    schema-generator.js   — entities[] → convex/schema.ts content
    data-seeder.js        — per-table JSONL conversion + npx convex import
    sql-type-map.js       — SQL type string → Convex validator string
    index.js              — exports
  package.json
```

### Dependency flow

```
cli → (postgres adapter for parsing)
    → @jerrythomas/dbd-convex (for apply/import when target=convex)
```

The Convex package has no dependency on the postgres adapter. It receives already-parsed entities from the CLI.

### Environment variables

| Variable            | Purpose                                               | Required                              |
| ------------------- | ----------------------------------------------------- | ------------------------------------- |
| `DATABASE_URL`      | PostgreSQL connection (DDL parsing, unchanged)        | For parsing/apply to postgres         |
| `CONVEX_URL`        | Convex deployment URL (`https://<name>.convex.cloud`) | For convex operations                 |
| `CONVEX_DEPLOY_KEY` | Auth token for deploy and seed operations             | For deploy/seed (not schema gen only) |

## Schema Generation (`apply --target=convex` / `dbd convex schema`)

### Input

Parsed table entities from the postgres adapter (same entities used for `dbd apply`).

### Output

`convex/schema.ts` written relative to the project root (same directory as `design.yaml`).

### Filtering

Only `table` entities are included. Views, procedures, functions, extensions, roles, and schemas are silently skipped — Convex has no equivalent constructs.

### SQL → Convex type mapping

| SQL types                                                                     | Convex validator        |
| ----------------------------------------------------------------------------- | ----------------------- |
| `text`, `varchar`, `char`, `citext`, `uuid`, `name`                           | `v.string()`            |
| `int`, `int2`, `int4`, `int8`, `bigint`, `smallint`, `serial`, `bigserial`    | `v.number()`            |
| `float4`, `float8`, `numeric`, `decimal`, `money`, `real`, `double precision` | `v.number()`            |
| `boolean`                                                                     | `v.boolean()`           |
| `json`, `jsonb`                                                               | `v.any()`               |
| `timestamp`, `timestamptz`, `date`, `time`, `timetz`                          | `v.string()` (ISO 8601) |
| `bytea`                                                                       | `v.bytes()`             |
| `<type>[]` (arrays)                                                           | `v.array(<inner type>)` |
| unknown                                                                       | `v.any()`               |

### Nullability

- Non-nullable column → bare validator: `v.string()`
- Nullable column → `v.optional(v.string())`

### Primary key handling

Columns that are part of the table's primary key constraint are dropped from the Convex schema — Convex auto-assigns `_id` as a system field. Primary key columns are identified via the `primaryKey` field on parsed table entities (set by the postgres parser).

### Table naming

Default: strip schema prefix. `public.users` → `users`, `config.features` → `features`.

Opt-in schema prefix via `design.yaml`:

```yaml
convex:
  schemaPrefix: true
  schemaPrefixSkip: [public] # default when schemaPrefix: true
```

With `schemaPrefix: true` and default skip list: `config.features` → `config_features`, `public.users` → `users`.

Collision handling: if two tables resolve to the same Convex table name after prefix resolution, emit a warning and suffix the second with `_<schema>` regardless of `schemaPrefixSkip`.

### Generated file format

```typescript
// convex/schema.ts — generated by dbd
import { defineSchema, defineTable } from 'convex/server'
import { v } from 'convex/values'

export default defineSchema({
  users: defineTable({
    name: v.string(),
    email: v.optional(v.string()),
    created_at: v.string()
  }),
  config_features: defineTable({
    key: v.string(),
    enabled: v.boolean()
  })
})
```

### Optional deploy

If `CONVEX_URL` and `CONVEX_DEPLOY_KEY` are both present, after writing the file dbd shells to:

```bash
npx convex deploy
```

If either env var is missing, schema generation succeeds but deploy is skipped with an info message.

### Dry-run

Prints the `schema.ts` content to stdout. Does not write the file or deploy.

## Data Seeding (`import --target=convex` / `dbd convex seed`)

### Input

The import plan (same as `dbd import`) — a list of tables with associated CSV or JSONL file paths.

### Per-table flow

1. Read the source file (CSV or JSONL)
2. For CSV: parse headers + rows → array of row objects; values coerced to match inferred types (numbers stay numbers, booleans as booleans, everything else as strings)
3. Write a temporary `.jsonl` file (one JSON object per line)
4. Shell to: `npx convex import --table <tableName> [--prod] <file.jsonl>`
5. Delete the temp file

`--prod` flag is appended when the `env` parameter passed to `importData` is `'prod'` (i.e. `dbd import -e prod --target=convex` or `dbd convex seed -e prod`).

The `tableName` uses the same naming logic as schema generation (respects `schemaPrefix` config).

### Auth

`CONVEX_DEPLOY_KEY` is required. The Convex CLI reads it automatically from the environment.

### Dry-run

Prints the `npx convex import` command and a sample of rows per table. Does not write files or shell out.

## CLI Changes

### New `--target` option on `apply` and `import`

```bash
dbd apply --target=convex
dbd apply --target=convex --dry-run

dbd import --target=convex
dbd import --target=convex --dry-run
dbd import --target=convex -n staging.users
```

The `--target` default remains unset (existing postgres behavior unchanged).

### New `dbd convex` subcommands

```bash
dbd convex schema              # equivalent to: dbd apply --target=convex
dbd convex schema --dry-run

dbd convex seed                # equivalent to: dbd import --target=convex
dbd convex seed --dry-run
dbd convex seed -n staging.users
```

Both subcommands share the same options as their `apply`/`import` equivalents (`-c`, `-n`, `--dry-run`, `-e`).

### Internal wiring

`design.apply(name, dryRun, target)` and `design.importData(name, dryRun, target)` gain a third `target` parameter (default: `undefined`). When `target === 'convex'`, they delegate to the convex package instead of the adapter.

## `design.yaml` config

```yaml
project:
  database: postgresql # unchanged — DDL parsing always uses postgres adapter

convex:
  schemaPrefix: false # opt-in: prefix table names with schema_
  schemaPrefixSkip: [public] # schemas exempt from prefixing (default)
```

The `convex:` section is optional. Defaults apply if absent.

## Testing

- `packages/convex` unit tests for:
  - `sql-type-map.js` — all type mappings including nullable/non-nullable and arrays
  - `schema-generator.js` — entity → schema.ts output, schemaPrefix variants, collision handling, PK stripping
  - `data-seeder.js` — CSV and JSONL parsing, temp file generation, dry-run output
- CLI integration tests for `--target=convex` and `dbd convex schema/seed` commands (dry-run mode, no live Convex deployment required)
- No E2E tests against a live Convex deployment (out of scope for CI)

---

## Out of Scope

- Convex indexes (no DDL index → Convex index mapping)
- Convex relationships / `v.id("table")` foreign key inference
- Schema push without the Convex CLI (no direct HTTP schema API)
- Full Convex adapter implementing `BaseDatabaseAdapter` (parsing Convex schemas back to SQL)
- Automatic rollback / import `--replace` flag (can be added later)
